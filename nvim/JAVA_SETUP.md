# NeoVim Java Development Setup

This NeoVim configuration validates Java code against your project's `checkstyle.xml` using **nvim-lint**.

## What's Configured

### Checkstyle Linting (via nvim-lint)
- Validates code against `checkstyle.xml` from project root
- Shows violations as inline diagnostics in the editor
- Auto-runs on file save
- Can be manually triggered with `<leader>ll`

### Formatting
- **NOT configured in NeoVim** - Formatting is done via your IDE
- Use **eclipse-formatter.xml** imported into IntelliJ/VS Code
- Import via your IDE's code style settings

## Keymaps

| Keymap | Action |
|--------|--------|
| `<leader>ll` | Manually run checkstyle linter |
| `gl` | Show diagnostic under cursor |
| `[d` | Go to previous diagnostic |
| `]d` | Go to next diagnostic |

## Workflow

1. Write/edit Java code
2. Save file (`Cmd+S` / `Ctrl+S`)
3. Checkstyle validates automatically → violations appear as red/yellow squiggles
4. **Format in your IDE** (IntelliJ Cmd+Option+L / VS Code Shift+Option+F)
5. Fix any remaining checkstyle violations manually
6. Save again

## Why This Approach?

- **No Maven plugins** - Just IDE and linter
- **No CI formatting** - Developers own formatting via their editors
- **Cross-IDE consistency** - eclipse-formatter.xml works everywhere
- **Real-time feedback** - NeoVim shows violations instantly
- **Simple setup** - Just import one config file per IDE

## Dependencies

Auto-installed by system:
- `checkstyle` - CLI tool for validation

Verify installation:
```bash
which checkstyle
```

## Configuration Files

- `checkstyle.xml` - Validation rules (found automatically in project root)
- `eclipse-formatter.xml` - Formatting config (import in your IDE)

See project README for import instructions per IDE.

# Auto-Generated Sources Configuration (jdtls)

When working with projects that have auto-generated code (OpenAPI, Protocol Buffers, gRPC, MapStruct, etc.), jdtls may not recognize the generated sources, causing false LSP errors like:

```
The import com.example.api.model cannot be resolved
```

Even though `mvn clean install` succeeds.

## The Issue

jdtls reads the `.classpath` file (generated by `mvn eclipse:eclipse`) to know where source files are located. If auto-generated sources aren't registered, jdtls can't find them even though:
- Maven compiles them fine
- They exist in `target/generated-sources/`
- IntelliJ/VS Code work (they have different classpath resolution)

## Two Solutions

### Solution 1: POM.xml Change (Recommended for Teams)

Add `build-helper-maven-plugin` to register generated sources:

```xml
<build>
  <plugins>
    <plugin>
      <groupId>org.codehaus.mojo</groupId>
      <artifactId>build-helper-maven-plugin</artifactId>
      <version>3.3.0</version>
      <executions>
        <execution>
          <id>add-generated-sources</id>
          <phase>generate-sources</phase>
          <goals>
            <goal>add-source</goal>
          </goals>
          <configuration>
            <sources>
              <source>${project.build.directory}/generated-sources/openapi/src/gen/java/main</source>
              <!-- Add other generated source directories as needed -->
            </sources>
          </configuration>
        </execution>
      </executions>
    </plugin>
  </plugins>
</build>
```

Then run: `mvn eclipse:eclipse`

**Pros:** Universal (works everywhere), committed to git, teammates get it automatically
**Cons:** Requires pom.xml changes

### Solution 2: jdtls Config (Neovim Only)

If you're only using Neovim and don't want pom.xml changes, modify `~/.config/nvim/lua/plugins/java.lua`:

```lua
{
  "mfussenegger/nvim-jdtls",
  opts = function(_, opts)
    local original_jdtls = opts.jdtls
    opts.jdtls = function(jdtls_opts, root_dir)
      if type(original_jdtls) == "function" then
        jdtls_opts = original_jdtls(jdtls_opts, root_dir)
      end
      
      -- Tell jdtls about generated source directories
      jdtls_opts.settings = vim.tbl_deep_extend("force", jdtls_opts.settings or {}, {
        java = {
          project = {
            sourcePaths = {
              "src/main/java",
              "src/test/java",
              "target/generated-sources/openapi/src/gen/java/main",
              "target/generated-sources/annotations",
            }
          }
        }
      })
      
      return jdtls_opts
    end
    
    return opts
  end,
}
```

**Pros:** No pom.xml changes, pure Neovim config
**Cons:** IDE-specific, won't help teammates, need to keep paths updated manually

Then restart jdtls: `:LspRestart`

## How to Know Which Generated Directories Exist

```bash
find <project>/target/generated-sources -type d -maxdepth 2
```

Common ones:
- `target/generated-sources/openapi/src/gen/java/main` (OpenAPI Generator)
- `target/generated-sources/annotations` (MapStruct, Lombok)
- `target/generated-sources/protobuf` (Protocol Buffers)
- `target/generated-sources/grpc` (gRPC)

## When to Use Which Solution

| Scenario | Solution |
|----------|----------|
| Working solo on personal project | Solution 2 (jdtls config) |
| Working with team | Solution 1 (pom.xml) |
| Mixed (want option to do both) | Solution 1 + jdtls config as backup |
| Team doesn't care about Neovim | Solution 2 (jdtls config only) |

## Troubleshooting

**Still seeing LSP errors after changes:**

1. Did you run `mvn clean install`? (to regenerate sources)
2. Did you run `mvn eclipse:eclipse`? (to regenerate .classpath if using Solution 1)
3. Did you restart jdtls? `:LspRestart` in Neovim
4. Check `.classpath` for your generated directories: `grep generated-sources ~/.classpath`

**Errors still there:**

Try clearing jdtls cache and workspace:
```bash
rm -rf ~/.cache/jdtls/workspace 2>/dev/null
rm -rf ~/.local/share/nvim/lsp.log 2>/dev/null
```

Then restart Neovim and give jdtls 30 seconds to re-index.
